# Модуль чеклистов: обзор, позиционирование и материалы для лендинга / FAQ

## 1. Что это за модуль

Модуль чеклистов в DashAdmin - это система стандартизации проверок, приемки смен и контроля качества внутри клуба.

Это не просто конструктор вопросов. Модуль связывает между собой:

- шаблоны проверок;
- сценарии аудита управляющего;
- приемку и сдачу смены;
- обязательные фото-доказательства;
- оценку по баллам;
- проверку рабочих мест по зонам;
- центр проверок;
- KPI и оценку качества работы сотрудников.

Идея модуля: проверка должна быть не "на словах", а в системе.
Не просто "управляющий посмотрел зал", а:

- по какому шаблону шла проверка;
- кого именно проверяли;
- какие пункты были пройдены;
- где были замечания;
- какие фото приложены;
- каков итоговый балл;
- была ли проверка принята или возвращена;
- влияет ли эта проверка на KPI и премию.

## 2. Для кого этот модуль

Модуль рассчитан на клубы, где важно держать единый стандарт состояния зала, смен и рабочих мест:

- компьютерные клубы;
- киберспортивные арены;
- сети клубов;
- площадки с несколькими сменами и несколькими управляющими;
- команды, где важно не только "сделано", но и "проверено по стандарту".

Основные роли внутри модуля:

- владелец клуба;
- управляющий;
- администратор;
- сотрудник смены;
- проверяющий или внутренняя функция контроля качества.

## 3. Какую проблему он решает

Без такого модуля чеклисты обычно живут в хаосе:

- часть пунктов в голове управляющего;
- часть в чате;
- часть в бумажках;
- часть в Notion / Excel;
- а часть каждый раз проверяется "по ощущениям".

Из-за этого возникают типовые проблемы:

- у разных управляющих разные стандарты;
- одна и та же смена принимается каждый раз по-разному;
- сотрудник не понимает, что именно считается ошибкой;
- фото не прикрепляются и потом нечем подтвердить замечание;
- сложно понять, где единичный косяк, а где системная проблема;
- результаты проверок не связаны с KPI;
- приемка смены формально есть, но реально не управляет процессом открытия и закрытия смены.

Модуль закрывает это за счет единой модели проверки:

- есть шаблон;
- у шаблона есть тип и настройки;
- у каждого пункта есть вес;
- замечания могут иметь свои варианты и свои баллы;
- можно требовать фото;
- можно привязывать проверку к рабочим местам и зонам;
- итог хранится как проверка с результатом, комментариями и статусом ревью.

## 4. Из чего состоит модуль

### 4.1. Список шаблонов чеклистов

Раздел `/settings/checklists` - это реестр всех шаблонов проверок клуба.

Что уже есть:

- список всех шаблонов;
- деление по типу шаблона;
- количество пунктов в шаблоне;
- переход в редактирование;
- создание нового шаблона;
- удаление шаблона.

Это важный момент для позиционирования: модуль начинается не с одноразовой формы, а с библиотеки стандартов клуба.

### 4.2. Конструктор шаблона

У каждого шаблона есть:

- название;
- описание;
- тип;
- настройки поведения;
- набор пунктов проверки.

Что умеет сам конструктор:

- добавлять и удалять пункты;
- менять порядок пунктов;
- задавать вес пункта в баллах;
- добавлять описание для проверяющего;
- включать обязательное фото;
- задавать минимальное количество фото;
- включать проверку рабочих мест;
- ограничивать проверку определенной зоной;
- добавлять варианты замечаний со своими баллами.

Это делает чеклист не просто списком "да / нет", а полноценным стандартом проверки с логикой оценки.

### 4.3. Два типа шаблонов

Сейчас в системе есть два базовых типа:

- `manager_audit` - аудит управляющего;
- `shift_handover` - приемка смены.

Это важно, потому что шаблоны используются не одинаково.

Аудит управляющего нужен для контроля качества.
Приемка смены встроена прямо в операционный сценарий сотрудника.

### 4.4. Настройки приемки смены

Для шаблонов `shift_handover` уже есть специальные настройки:

- блокировать открытие смены без прохождения чеклиста;
- блокировать закрытие смены без прохождения чеклиста.

Это один из самых сильных смыслов модуля.
Чеклист перестает быть "рекомендацией" и становится частью рабочего процесса.

### 4.5. Настройки аудита

Для шаблонов `manager_audit` уже есть настройка:

- влияет ли результат проверки на KPI.

Это важно для бизнеса:

- качество начинает измеряться не интуитивно;
- проверка становится частью системы мотивации;
- у клуба появляется воспроизводимый стандарт оценки.

### 4.6. Пункты с проверкой рабочих мест

Отдельная сильная часть текущей реализации - пункты, которые работают не абстрактно, а по рабочим местам.

Что это дает:

- можно проверять не "зал в целом", а конкретные места;
- можно ограничить пункт определенной зоной;
- можно фиксировать проблемные рабочие места прямо внутри чеклиста;
- итоговая проверка становится ближе к реальной операционке клуба.

Для коммуникации это сильный тезис: чеклист умеет проверять не только человека, но и конкретную физическую среду.

### 4.7. Варианты замечаний и балльная логика

Для пункта можно задать не только общий вес, но и конкретные варианты замечаний.

Например:

- "идеально" - 5;
- "есть мелкие разводы" - 4;
- "заметные пятна" - 3;
- "грязно" - 0.

Это делает оценку:

- более объективной;
- менее зависимой от настроения проверяющего;
- удобной для обучения новых управляющих;
- пригодной для KPI и аналитики.

### 4.8. Проведение проверки

Чеклист используется не только в настройках, но и в реальном процессе проведения проверки.

Что уже есть:

- выбор шаблона;
- выбор сотрудника;
- привязка к смене;
- заполнение ответов по пунктам;
- комментарии;
- фото;
- сохранение итогового результата в оценку.

Итог сохраняется как реальная проверка, а не как временная форма.

### 4.9. Центр проверок

После прохождения проверки она живет дальше в модуле `reviews`.

Там уже есть:

- активные проверки;
- история проверок;
- статус на ревью;
- возврат с комментарием;
- одобрение;
- просмотр деталей конкретной проверки.

Это важный продуктовый смысл:
модуль покрывает не только создание чеклистов, но и полный цикл "провели -> проверили -> приняли / вернули".

### 4.10. Связь со сменами

Чеклисты уже встроены в shift-flow:

- шаблон приемки может блокировать старт смены;
- шаблон приемки может блокировать завершение смены;
- результаты чеклистов видны внутри карточки смены;
- проверки могут быть привязаны к конкретной смене.

Это делает модуль частью ядра сменной операционки, а не отдельным инструментом "на всякий случай".

## 5. Ключевая бизнес-ценность

### 5.1. Для собственника и управляющего

- У клуба появляется единый стандарт проверки.
- Видно, кто и как проходит проверки.
- Можно убрать субъективность из приемки смены.
- Проще выявлять повторяющиеся проблемы по залу, смене и сотруднику.
- Проверки можно завязывать на KPI и мотивацию.

### 5.2. Для операционной команды

- Сотрудникам понятнее, что именно от них ожидается.
- Снижается количество споров формата "я не знал, что это нужно проверять".
- Приемка и сдача смены становятся структурированными.
- Фото и комментарии фиксируются сразу в контексте проверки.

### 5.3. Для качества сервиса

- Поддерживается единое состояние клуба между сменами.
- Проблемы фиксируются раньше, а не после жалоб гостей.
- Стандарты можно масштабировать между филиалами и управляющими.
- Качество становится измеримым, а не только обсуждаемым.

## 6. Что особенно сильное в этой реализации

### 6.1. Это не просто шаблон формы, а управленческий стандарт

Шаблон здесь - это не "сделали список вопросов".
Это рабочее правило клуба, которое потом реально используется в операционке.

### 6.2. Чеклисты встроены в открытие и закрытие смены

Это один из самых сильных differentiator'ов:

- шаблон может блокировать открытие смены;
- шаблон может блокировать закрытие смены;
- значит, стандарт реально влияет на процесс, а не висит отдельно.

### 6.3. Есть балльная логика и варианты замечаний

Это делает оценки более точными.
Вместо грубого "норм / не норм" клуб получает управляемую шкалу качества.

### 6.4. Есть проверка рабочих мест, а не только сотрудника

Система умеет смотреть на клуб как на физическую среду:

- какие места проверялись;
- в какой зоне нашли проблемы;
- где именно требуется внимание.

### 6.5. Есть фото-доказательства

Фото и комментарии не живут отдельно в чате.
Они хранятся прямо в контексте конкретного пункта проверки.

### 6.6. Результаты уже связаны с ревью и KPI

Проверка не заканчивается на кнопке "отправить".
Она дальше живет в:

- центре проверок;
- истории проверок;
- оценке качества;
- зарплатных и KPI-сценариях.

## 7. Основные сценарии использования

### Сценарий 1. Настройка стандарта клуба

1. Управляющий создает шаблон.
2. Выбирает тип: аудит или приемка смены.
3. Добавляет пункты проверки.
4. Настраивает баллы, фото и варианты замечаний.
5. Сохраняет шаблон как стандарт клуба.

### Сценарий 2. Приемка открытия смены

1. Для клуба настроен шаблон `shift_handover`.
2. Включена блокировка открытия смены.
3. Сотрудник пытается начать смену.
4. Система сначала требует пройти чеклист.
5. Только после этого смена может быть открыта.

### Сценарий 3. Приемка закрытия смены

1. Для шаблона включена блокировка закрытия смены.
2. Сотрудник завершает смену.
3. Система требует пройти обязательную приемку.
4. При необходимости прикладываются фото и отмечаются проблемные рабочие места.
5. После прохождения разрешается завершение смены.

### Сценарий 4. Аудит управляющего

1. Управляющий выбирает шаблон аудита.
2. Выбирает сотрудника и при необходимости смену.
3. Заполняет пункты, комментарии и фото.
4. Отправляет проверку.
5. Результат попадает в центр проверок и историю.

### Сценарий 5. Анализ качества работы

1. Проверки накапливаются в системе.
2. Видны активные и исторические оценки.
3. Результаты можно использовать для KPI.
4. Управляющий видит, где проблема системная, а где разовая.

## 10. FAQ-заготовки

### Что умеет модуль чеклистов?

Модуль позволяет создавать шаблоны проверок, настраивать пункты и баллы, требовать фото, проверять рабочие места, использовать чеклисты в приемке смены и отправлять результаты в центр проверок.

### Это просто конструктор форм?

Нет. Шаблоны здесь встроены в операционный процесс клуба. Они могут участвовать в открытии и закрытии смены, проходить ревью и влиять на KPI.

### Какие типы чеклистов есть?

Сейчас есть как минимум два базовых типа: аудит управляющего и приемка смены.

### Можно ли блокировать открытие смены, пока чеклист не пройден?

Да. Для шаблона приемки можно включить блокировку открытия смены.

### Можно ли блокировать закрытие смены?

Да. Для шаблона приемки можно включить блокировку закрытия смены.

### Можно ли требовать фото по пункту?

Да. Для конкретного пункта можно включить обязательное фото и даже задать минимальное количество фото.

### Можно ли оценивать не только "да / нет"?

Да. Для пункта можно задать балльный вес и варианты замечаний со своими значениями.

### Можно ли использовать чеклисты для проверки рабочих мест?

Да. Пункты могут работать по рабочим местам и при необходимости быть ограничены определенной зоной клуба.

### Где потом видны результаты проверки?

Результаты видны в центре проверок, в истории и в карточке смены, если проверка привязана к смене.

### Влияют ли чеклисты на KPI?

Да, для аудиторских шаблонов уже есть сценарий, где результат может влиять на KPI.

### Подходит ли модуль только для смен?

Нет. Приемка смены - только один из сценариев. Второй крупный сценарий - аудиты управляющего и контроль стандартов качества по клубу.
