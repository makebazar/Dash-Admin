# Модуль оборудования: обзор, позиционирование и материалы для лендинга / FAQ

## 1. Что это за модуль

Модуль оборудования в DashAdmin - это операционная система для учета, размещения, обслуживания и контроля техники внутри клуба.

Это не просто список устройств. Модуль связывает между собой:

- реестр оборудования;
- зоны и рабочие места;
- инциденты и ремонты;
- плановое обслуживание;
- назначение ответственных;
- стирку ковриков и возврат;
- инструкции и стандарты обслуживания по типам техники.

Идея модуля: у каждой единицы техники есть контекст.
Не просто "есть мышка Logitech", а:

- где она стоит;
- в каком она статусе;
- кто за нее отвечает;
- когда ее обслуживали в последний раз;
- есть ли по ней инциденты;
- есть ли по ней текущая задача;
- требуется ли стирка/возврат;
- по какому стандарту сотрудники должны ее обслуживать.

## 2. Для кого этот модуль

Модуль рассчитан на клубы и площадки, где техника постоянно участвует в операционке:

- компьютерные клубы;
- киберспортивные арены;
- VR / gaming spaces;
- площадки с большим количеством рабочих мест и периферии;
- франчайзинговые клубы, где нужен единый стандарт обслуживания.

Основные роли внутри модуля:

- владелец или управляющий клуба;
- администратор;
- техперсонал;
- сотрудник, выполняющий задачи по обслуживанию;
- центр проверок или внутренняя функция контроля качества.

## 3. Какую проблему он решает

Без такого модуля оборудование обычно живет в пяти местах одновременно:

- часть в голове управляющего;
- часть в Excel;
- часть в переписке;
- часть в устных договоренностях;
- часть вообще нигде.

Из-за этого появляются типовые проблемы:

- непонятно, сколько техники реально есть в клубе;
- неясно, где конкретно стоит каждое устройство;
- инциденты теряются или не доходят до исполнения;
- обслуживание делается нерегулярно;
- ответственные назначены "словами", но не закреплены в системе;
- задачи уходят не тем людям;
- нельзя быстро показать, что сломано, что просрочено и где узкое место;
- новые сотрудники работают без единого стандарта.

Модуль закрывает это за счет единой операционной модели:

- техника привязана к месту;
- место привязано к зоне;
- задачи и инциденты привязаны к конкретным устройствам;
- ответственность хранится явно;
- график чистки и контроль просрочки считаются автоматически;
- фото и обсуждение хранятся рядом с проблемой;
- стандарты обслуживания задаются на уровне типа оборудования.

## 4. Из чего состоит модуль

### 4.1. Главная страница оборудования

Главная страница модуля показывает два ключевых сигнала:

- открытые инциденты;
- просроченное обслуживание.

И отдельно ведет в основные разделы:

- рабочие места;
- инвентаризация;
- инциденты;
- график чистки;
- настройки;
- стирка.

Это важно для позиционирования: модуль начинается не с таблицы, а с управленческих сигналов и быстрых сценариев действий.

### 4.2. Рабочие места

Раздел рабочих мест отвечает за физическую структуру клуба:

- зоны;
- места внутри зон;
- оборудование, установленное на каждом месте.

Что умеет раздел:

- показывать структуру клуба по зонам;
- видеть пустые места и проблемные места;
- открывать детали конкретного места;
- назначать и снимать оборудование с места;
- видеть просрочку обслуживания по каждому устройству;
- видеть активные проблемы по месту.

Главная ценность раздела: он связывает абстрактный инвентарь с реальной географией клуба.

### 4.3. Инвентаризация

Это единый реестр оборудования и периферии.

Что уже есть:

- поиск;
- фильтры по типу, зоне, месту, статусу;
- группировка по местам;
- статусы техники;
- гарантия;
- признак проблем;
- экспорт в XLSX;
- шаблон для импорта;
- импорт из XLSX / XLS / CSV;
- обновление существующих записей по ID или серийному номеру;
- создание новых записей;
- базовые массовые действия.

Что важно для коммуникации:

- импорт не создает зоны и места автоматически;
- сначала структура клуба, потом массовая загрузка техники;
- это снижает хаос и повышает качество данных.

### 4.4. Инциденты

Раздел инцидентов - это система фиксации проблем, ремонта и коммуникации по оборудованию.

Что уже покрыто:

- создание инцидента по конкретному устройству;
- группировка по месту;
- приоритет;
- статус;
- назначение ответственного;
- отдельная детальная страница инцидента;
- обсуждение в комментариях;
- фото при создании;
- фото решения;
- подтягивание фото, если инцидент вырос из задачи обслуживания.

Это не просто тикеты "в общем". Каждый инцидент привязан к конкретному устройству и месту, поэтому по нему легче понять контекст и быстрее принять решение.

### 4.5. Обслуживание

Это один из самых сильных разделов модуля.

Он отвечает за:

- плановую чистку и регламентные работы;
- распределение задач по сотрудникам;
- контроль просрочки;
- контроль доработок после проверки;
- перераспределение нагрузки;
- видимость свободного пула.

Ключевая логика:

- задачи формируются по графику;
- ответственным может быть конкретный сотрудник или свободный пул;
- место может получить ответственного, и это назначение применяется ко всему оборудованию на месте;
- отдельное устройство можно переназначить отдельно;
- задачи не должны уходить неактивным сотрудникам;
- в назначение попадают только активные сотрудники по графику и управляющий / manager-сценарии;
- система показывает, где просрочка, что в работе, что на доработке и что запланировано позже.

Для лендинга это можно подавать как "система эксплуатации оборудования", а не как "таблица чистки".

### 4.6. Мои задачи

Есть отдельный employee-side сценарий:

- сотрудник видит только свои задачи;
- может начать выполнение;
- может завершить задачу;
- видит просрочку и текущий план.

Это делает модуль не только управленческим, но и исполнительно-операционным.

### 4.7. Стирка

Отдельный сценарий для ковриков и схожих расходных единиц ухода:

- новая заявка;
- отправлено в стирку;
- готово к возврату;
- возвращено;
- отменено.

Стирка уже связана с общим модулем оборудования и с историей рабочего места.
Это важный нюанс для лендинга: модуль покрывает не только "железо", но и операционные процессы вокруг него.

### 4.8. Настройки и стандарты

В модуле есть база стандартов обслуживания:

- инструкции по типам оборудования;
- интервал обслуживания по типу;
- зоны;
- типы оборудования.

Это позволяет сделать обслуживание масштабируемым:

- новый сотрудник видит не устные договоренности, а стандарт;
- клуб может формализовать свои регламенты;
- можно поддерживать единый подход между сменами и филиалами.

## 5. Ключевая бизнес-ценность

### 5.1. Для собственника и управляющего

- Понятно, сколько техники реально в работе, на складе, в ремонте и в списании.
- Видно, где копятся проблемы и просрочка.
- Можно быстро оценить, кто перегружен, а где задачи висят в свободном пуле.
- Снижается зависимость от конкретного администратора "который и так все помнил".

### 5.2. Для операционной команды

- У каждого устройства есть понятный контекст.
- Инциденты не теряются.
- Плановая чистка перестает жить в голове.
- Назначение ответственных становится явным, а не "договорились устно".

### 5.3. Для качества сервиса

- Меньше грязной или проблемной техники на местах.
- Быстрее цикл от обнаружения проблемы до решения.
- Легче удерживать единый стандарт состояния клуба.
- Появляется доказательная база: фото, комментарии, даты, статусы.

## 6. Что особенно сильное в этой реализации

Это то, что реально можно выносить в коммуникацию.

### 6.1. Модуль построен вокруг места, а не вокруг абстрактного списка

Большинство систем учета ограничиваются списком устройств.
Здесь техника живет в структуре клуба:

- зона;
- место;
- устройство;
- инцидент / задача / история.

Это сильно ближе к реальной эксплуатации.

### 6.2. Ответственность закрепляется явно

Это важное отличие текущей логики:

- если назначен ответственный за место, это не "виртуальное наследование";
- система реально проставляет ответственного оборудованию этого места;
- отдельное устройство можно переопределить отдельно;
- если ответственный снят, устройство уходит в свободный пул.

Для FAQ это очень хороший пункт, потому что он отвечает на частый вопрос "а кто в итоге отвечает - место или устройство?".

### 6.3. Задачи распределяются не просто по списку сотрудников

Система учитывает операционный контекст:

- активность сотрудника;
- наличие в графике;
- исключение для управляющего / manager-ролей;
- свободный пул как осознанный сценарий, а не ошибка.

### 6.4. Инциденты и обслуживание связаны

Проблема по оборудованию не оторвана от его жизненного цикла.
Фото при создании проблемы могут прийти из обслуживания, а результат решения хранится там же.

### 6.5. Есть стандарты, а не только фиксация факта

Модуль не только показывает "что произошло", но и хранит "как должно делаться правильно".

## 7. Основные сценарии использования

### Сценарий 1. Запуск клуба или нового зала

1. Создаются зоны.
2. Внутри зон создаются рабочие места.
3. Через импорт загружается техника.
4. Оборудование привязывается к местам.
5. Для типов задаются стандарты и интервалы обслуживания.

### Сценарий 2. Ежедневная операционка

1. Администратор или сотрудник замечает проблему.
2. Создается инцидент по устройству.
3. Назначается ответственный.
4. По инциденту ведется обсуждение и прикладываются фото.
5. После решения фиксируется результат и тикет закрывается.

### Сценарий 3. Плановое обслуживание

1. Система формирует задачи на период.
2. Задачи распределяются по ответственным.
3. Сотрудники выполняют обслуживание.
4. Просрочка и доработки видны управляющему.
5. Следующие задачи пересчитываются по факту выполнения.

### Сценарий 4. Изменение комплектации места

1. Устройство снимается с места или переносится.
2. Оно уходит на склад или на новое место.
3. Контекст места обновляется.
4. Новое распределение задач и ответственности строится уже по актуальной структуре.

### Сценарий 5. Работа с ковриками / стиркой

1. Коврик попадает в очередь.
2. Система проводит его по статусам.
3. Возврат фиксируется.
4. Событие видно в истории рабочего места.

## 10. FAQ-заготовки

### Что умеет модуль оборудования?

Модуль ведет реестр техники, показывает расположение по зонам и местам, фиксирует инциденты, управляет плановым обслуживанием, назначает ответственных, поддерживает импорт/экспорт и хранит стандарты обслуживания по типам оборудования.

### Это просто инвентаризация?

Нет. Инвентаризация - только одна часть модуля. Помимо учета, система покрывает ежедневную эксплуатацию техники: проблемы, ремонт, обслуживание, назначение сотрудников и связанные операционные процессы.

### Можно ли привязать оборудование к конкретным местам?

Да. Техника размещается по рабочим местам внутри зон клуба. Это позволяет видеть не только список устройств, но и их фактическое расположение в зале.

### Можно ли назначать ответственных?

Да. Ответственного можно назначить на рабочее место или на конкретное устройство. При назначении на место система применяет это назначение ко всему оборудованию на этом месте. При необходимости отдельное устройство можно переопределить отдельно.

### Что происходит, если ответственный не назначен?

Оборудование может оставаться в свободном пуле. Это осознанный сценарий: система показывает, где задачи не закреплены за конкретным человеком.

### Как работает плановое обслуживание?

Для типов оборудования задаются стандарты и интервалы. Система формирует задачи по графику, показывает просрочку, статус выполнения и распределение задач между сотрудниками.

### Можно ли распределять задачи по сотрудникам автоматически?

Система уже строит распределение на основе явных назначений по оборудованию и месту, а также учитывает активность сотрудников и их участие в графике. Это помогает избежать ситуаций, когда задачи назначаются не тем людям.

### Есть ли контроль качества выполнения?

Да. В задачах обслуживания есть сценарий доработки после проверки. Это позволяет отделять "выполнено" от "принято без замечаний".

### Можно ли фиксировать поломки с фото?

Да. Инциденты поддерживают фото при создании и фото после решения. Если проблема возникла из обслуживания, связанный визуальный контекст тоже можно показать.

### Есть ли импорт оборудования из Excel?

Да. Поддерживаются импорт и экспорт, а также шаблон для загрузки. Перед импортом нужно заранее создать зоны и рабочие места, чтобы данные ложились в корректную структуру клуба.

### Можно ли использовать модуль для сети клубов?

Да, особенно если важны единые стандарты обслуживания, контроль качества и одинаковая операционная модель между площадками.

### Подходит ли модуль только для ПК?

Нет. Модуль работает как с компьютерами, так и с периферией и другими типами оборудования, для которых можно задать собственные стандарты обслуживания и правила учета.

### Есть ли отдельный интерфейс для исполнителей?

Да. У сотрудников есть сценарий просмотра своих задач по обслуживанию и перевода их в работу / завершение.

### Зачем отдельно нужен раздел стирки?

Потому что часть клубного инвентаря требует не ремонта, а операционного цикла ухода и возврата. Стирка встроена в модуль как отдельный рабочий процесс, а не как заметка в комментариях.

## 12. Что лучше пока не обещать как готовую фичу

Это важно, чтобы лендинг не убежал дальше продукта.

- полноценную продвинутую аналитику по стоимости оборудования;
- сложные массовые сценарии ремонта как finished feature;
- глубокую автоматизацию уведомлений как уже готовый и настроенный блок;
- полноценный публичный self-service для всех ролей без оговорок;
- "AI-диагностику" или предиктивное обслуживание;
- полноценный модуль истории как центральный сценарий продукта.

Если это и упоминать, то только как roadmap / дальнейшее развитие.
