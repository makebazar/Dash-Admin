# Документация по интеграции CloudPayments

Данный документ содержит полное структурированное руководство по интеграции платежного шлюза **CloudPayments**. Он подготовлен на основе официальной технической документации и содержит описание схем взаимодействия, параметров платежного виджета, платежного конструктора, Checkout-скрипта, методов API, настройки уведомлений (webhooks) и фискализации чеков (54-ФЗ).

---

## 1. Общая информация и базовые понятия

### Термины и определения
* **Система** — платежный шлюз CloudPayments.
* **ТСП** (Торгово-сервисное предприятие) — продавец/мерчант, интегрирующий систему.
* **API** — программный интерфейс шлюза: `https://api.cloudpayments.ru`.
* **ЛК** — личный кабинет мерчанта: `https://merchant.cloudpayments.ru`.
* **Виджет** — готовая всплывающая безопасная платежная форма (iframe).
* **3-D Secure** — двухфакторный протокол проверки подлинности держателя карты банком-эмитентом (подтверждение через СМС/Push-код).

### Схемы проведения платежей
1. **Одностадийная (SMS — Single Message System)**
   * Списание средств происходит мгновенно после успешной авторизации.
   * Применяется для простых моментальных покупок.
2. **Двухстадийная (DMS — Dual Message System)**
   * Состоит из двух шагов: **Авторизация** (блокировка/холдирование суммы на карте до 7 дней) и **Подтверждение** (фактическое списание средств).
   * Если подтверждение не отправлено в течение установленного срока, деньги автоматически разблокируются.
   * Позволяет подтверждать списание меньшей суммы, чем исходно заблокированная, либо полностью отменять операцию без комиссий за эквайринг (удобно для отелей, аренды авто, проверки баланса).

## 2. Платежный виджет (CloudPayments Widget)

Платежный виджет представляет собой готовую всплывающую форму для ввода карточных данных. Он автоматически определяет тип платежной системы (Visa, MasterCard, МИР, Maestro), банк-эмитент, оптимизирован под десктопные и мобильные устройства, и работает внутри безопасного iframe.

### Установка виджета

Добавьте скрипт в раздел `<head>` вашего сайта:
```html
<script src="https://widget.cloudpayments.ru/bundles/cloudpayments.js"></script>
```
> [!IMPORTANT]
> Для соблюдения требований PCI DSS скрипт должен загружаться строго с домена `widget.cloudpayments.ru`. Не размещайте файл локально на своих серверах.

### Пример запуска виджета (на чистом JavaScript)

```javascript
const btn = document.getElementById("payButton");
const widget = new cp.CloudPayments();

const launchWidget = () => {
  const intentParams = {
    publicTerminalId: "test_api_00000000000000000000002", // Ваш Public ID из ЛК
    description: "Оплата заказа №1024",
    amount: 1000.00,
    currency: "RUB",
    paymentSchema: "Single", // 'Single' или 'Dual'
    invoiceId: "order_1024", // Номер заказа в ТСП
    accountId: "user_customer_99", // ID пользователя (для рекуррентов/токенизации)
    email: "customer@example.com",
    skin: "modern", // 'classic' (светлая) или 'modern' (темная)
    autoClose: 3, // Закрыть виджет автоматически через 3 секунды после успеха
    
    // Передача чека (CloudKassir 54-ФЗ)
    receipt: {
      items: [
        {
          label: "Подписка на сервис",
          price: 1000.00,
          quantity: 1.00,
          amount: 1000.00,
          vat: 20, // НДС 20%
          method: 0, // Способ расчета: Предоплата 100%
          object: 4 // Предмет расчета: Услуга
        }
      ],
      taxationSystem: 0, // ОСН
      email: "customer@example.com"
    }
  };

  widget.start(intentParams)
    .then((widgetResult) => {
        console.log("Успешный платеж:", widgetResult);
        // widgetResult содержит: transactionId, amount, status и т.д.
    })
    .catch((error) => {
        console.error("Ошибка при оплате:", error);
    });
};

btn.addEventListener("click", launchWidget);
```

### Основные параметры виджета

| Параметр | Тип | Описание | Обязательность |
| :--- | :--- | :--- | :--- |
| `publicTerminalId` | string | Публичный ID терминала из ЛК | **Да** |
| `amount` | float | Сумма платежа (должна быть > 0) | **Да** |
| `currency` | string | Валюта (`RUB`, `USD`, `EUR` и др. — см. справочник) | **Да** |
| `paymentSchema` | string | Схема оплаты: `Single` (одностадийная) или `Dual` (двухстадийная) | **Да** |
| `description` | string | Описание назначения платежа | Нет |
| `invoiceId` | string | Идентификатор заказа в вашей системе (ранее `externalId`) | Нет |
| `accountId` | string | Идентификатор плательщика (обязателен для получения токена привязки карты) | Нет |
| `email` | string | Email для отправки чека шлюзом | Нет |
| `restrictedPaymentMethods` | array | Список отключенных методов оплаты (`Card`, `Sbp`, `TinkoffPay`, `MirPay`, `Dolyame`, `SberPay`) | Нет |
| `tokenize` | boolean | Сохранить ли карту для последующих рекуррентных списаний | Нет |
| `receipt` | object | Объект чека для фискализации (CloudKassir) | Нет |
| `userInfo` | object | Детальные данные плательщика (телефон, ФИО, адрес для MCC 4814) | Нет |
| `metadata` | object | Любые кастомные JSON-данные (придут обратно в Webhook) | Нет |
| `skin` | string | Тема виджета: `classic` (светлая) или `modern` (темная) | Нет |

## 3. Платежный конструктор (PaymentBlocks)

Платежный конструктор позволяет встраивать элементы ввода реквизитов карты (номер, срок действия, CVC) непосредственно в интерфейс вашего сайта без всплывающих окон, сохраняя при этом безопасность уровня PCI DSS за счет использования защищенных iframe для каждого поля ввода.

### Основные отличия от виджета
* Кнопка оплаты находится на вашей странице.
* Вы можете полностью кастомизировать дизайн полей ввода (шрифты, рамки, цвета, плейсхолдеры).
* Поддерживает коллбэки на изменение состояния валидации полей, фокус и ошибки.

### Пример инициализации

```javascript
// Конфигурация стилей полей ввода внутри iframe
const customStyles = {
  input: {
    "font-size": "16px",
    "color": "#333333",
    "font-family": "sans-serif"
  },
  ".invalid": {
    "color": "red"
  }
};

const payments = new cp.PaymentBlocks({
  publicId: "test_api_00000000000000000000002",
  // Селекторы блоков в вашем HTML, куда встроятся iframe-поля
  cardNumberContainer: "#cardNumber",
  cardDateContainer: "#cardDate",
  cardCvcContainer: "#cardCvc",
  styles: customStyles
});

// Подписка на события валидации
payments.addEventListener("validationChange", (event) => {
  if (event.isValid) {
    document.getElementById("submitBtn").removeAttribute("disabled");
  } else {
    document.getElementById("submitBtn").setAttribute("disabled", "true");
  }
});
```

## 4. Технология Checkout (Кастомные формы ввода)

Если мерчант сертифицирован по стандарту PCI DSS (лист самооценки SAQ A-EP или выше) и проходит ASV-сканирование, он может использовать технологию **Checkout**. Это позволяет собирать карточные данные в полностью кастомную HTML-форму мерчанта.

При отправке формы специальный Checkout-скрипт от CloudPayments шифрует введенные данные карты прямо в браузере клиента в безопасную криптограмму (tokenized payload).
Затем эта криптограмма отправляется на сервер мерчанта, а сервер проводит платеж по API.

### Пример шифрования в JS

```javascript
// Подключение библиотеки шифрования
// <script src="https://widget.cloudpayments.ru/bundles/checkout.js"></script>

const checkout = new cp.Checkout({
  publicId: "YOUR_PUBLIC_ID"
});

const createCryptogram = () => {
  const cardData = {
    cardNumber: document.getElementById("card-number-input").value,
    cardDate: document.getElementById("card-date-input").value, // MM/YY
    cardCvc: document.getElementById("card-cvc-input").value,
    cardHolderName: document.getElementById("card-holder-input").value
  };

  checkout.createCryptogram(cardData)
    .then((cryptogram) => {
        console.log("Криптограмма успешно создана:", cryptogram);
        // Отправьте cryptogram на ваш сервер для проведения платежа через API
    })
    .catch((errors) => {
        console.error("Ошибки валидации карты:", errors);
    });
};
```

## 5. CloudPayments API

Базовый адрес API: `https://api.cloudpayments.ru`

### Аутентификация запросов
Для выполнения серверных запросов к API используется HTTP Basic Authentication:
* **Имя пользователя (Username):** Ваш `Public ID` (из личного кабинета).
* **Пароль (Password):** Ваш `API Secret` (из личного кабинета).

> [!IMPORTANT]
> Передавайте заголовок `Content-Type: application/json` для всех запросов.

### Идемпотентность API
Для предотвращения дублирования транзакций (например, при сетевых сбоях) поддерживается заголовок **`X-Request-ID`** (уникальный UUID запроса). При повторном запросе с тем же `X-Request-ID` в течение 1 часа шлюз вернет результат исходной операции без повторного списания денег.

### 1. Тестовый метод (Проверка связи)
* **Метод:** `POST`
* **Эндпоинт:** `/test`
* **Описание:** Проверяет корректность авторизации. Не требует параметров.

### 2. Оплата по криптограмме (созданной в Checkout или Mobile SDK)
* **Метод:** `POST`
* **Эндпоинты:** 
  * `/payments/cards/charge` (Одностадийная оплата)
  * `/payments/cards/auth` (Двухстадийная авторизация)
* **Основные параметры запроса:**

```json
{
  "Amount": 1000.00,
  "Currency": "RUB",
  "IpAddress": "192.168.1.1",
  "CardCryptogramPacket": "cryptogram_from_checkout_or_sdk...",
  "Name": "CARDHOLDER NAME",
  "InvoiceId": "order_1024",
  "AccountId": "user_customer_99",
  "Email": "customer@example.com",
  "JsonData": {
    "cloudpayments": {
      "recurrent": { "interval": "Month", "period": 1 } // Опционально для автоплатежей
    }
  }
}
```

* **Сценарии ответов:**
  * **Успешно проведен без 3-D Secure:** Возвращается `"Success": true` и объект `"Model"` со статусом `"Completed"` или `"Authorized"`.
  * **Требуется 3-D Secure проверка:** Возвращается `"Success": false` и параметры `"TransactionId"`, `"AcsUrl"`, `"PaReq"`. Необходимо перенаправить пользователя на `AcsUrl` с параметрами `PaReq`, `MD` (равным `TransactionId`) и `TermUrl` (адрес на вашем сервере, куда вернется клиент после ввода кода подтверждения).
  * **Отклонен банком:** Возвращается `"Success": false`, а в `"Model"` заполнены поля `"ReasonCode"` и `"CardHolderMessage"`.

### 3. Завершение оплаты после 3-D Secure (POST 3DS)
После прохождения 3DS на стороне банка-эмитента клиент перенаправляется на ваш `TermUrl` с параметрами `PaRes` и `MD`.
Вы должны отправить POST-запрос с вашего сервера для завершения авторизации:
* **Метод:** `POST`
* **Эндпоинт:** `/payments/cards/post3ds`
* **Параметры:**
  * `TransactionId` (значение `MD`)
  * `PaRes` (ответ от банка)

### 4. Оплата по токену (Рекарринг / Оплата в один клик)
Если при первом успешном платеже вы передали `AccountId`, в webhook придет параметр `Token` привязанной карты. Вы можете производить повторные списания без ввода реквизитов и 3-D Secure.
* **Метод:** `POST`
* **Эндпоинты:**
  * `/payments/tokens/charge` (Одностадийная по токену)
  * `/payments/tokens/auth` (Двухстадийная по токену)
* **Параметры:** `Token`, `Amount`, `Currency`, `AccountId`, `InvoiceId` (необязательно).

### 5. Подтверждение оплаты (для двухстадийных платежей)
* **Метод:** `POST`
* **Эндпоинт:** `/payments/confirm`
* **Параметры:** `TransactionId`, `Amount` (может быть меньше или равна сумме холдирования).

### 6. Отмена оплаты (для двухстадийных платежей в статусе Authorized)
* **Метод:** `POST`
* **Эндпоинт:** `/payments/void`
* **Параметры:** `TransactionId`. Средства мгновенно разблокируются на карте держателя.

### 7. Возврат средств (для завершенных платежей в статусе Completed)
* **Метод:** `POST`
* **Эндпоинт:** `/payments/refund`
* **Параметры:** `TransactionId`, `Amount` (поддерживается частичный возврат).

### 8. Выплаты на карты (Payout)
* **Метод:** `POST`
* **Эндпоинты:**
  * `/payments/cards/topup` (По криптограмме карты)
  * `/payments/token/topup` (По токену карты)
* **Параметры:** `Amount`, `Currency` (`RUB`), `AccountId`, `Token` / `CardCryptogramPacket`.
* **Дополнительно:** Позволяет ТСП выплачивать деньги клиентам (например, выигрыши, переводы). Требует согласования и настройки специального терминала выплат.

### 9. Проверка статуса транзакции
* **Метод:** `POST`
* **Эндпоинты:**
  * `/payments/get` — Детализация по `TransactionId`.
  * `/v2/payments/find` — Поиск последней транзакции по `InvoiceId` (ищет по оплатам, возвратам и выплатам).

## 6. Уведомления (Webhooks / Callbacks)

Уведомления — это входящие POST-запросы со стороны CloudPayments на ваши скрипты-обработчики (webhooks) при изменении статусов транзакций. Настраиваются в Личном Кабинете мерчанта.

### Безопасность и проверка подписи (HMAC-SHA256)
Каждое уведомление содержит заголовок **`Content-HMAC`** (подпись в кодировке Base64). 
Для проверки подлинности запроса необходимо вычислить HMAC-SHA256 от тела запроса (сырого JSON/строки), используя ваш **`API Secret`** в качестве секретного ключа, и сравнить результат со значением заголовка. Если они не совпадают — запрос должен быть отклонен (ошибка `401 Unauthorized`).

### Основные типы уведомлений

1. **`Check`**
   * Отправляется **до** проведения списания/авторизации, сразу после ввода реквизитов карты.
   * **Назначение:** Контроль возможности принять платеж (проверить наличие товара, сверить сумму заказа, валидировать `InvoiceId` в вашей БД).
   * **Ожидаемый ответ:** Мерчант должен вернуть JSON с кодом ответа:
     * `{"code": 0}` — Платеж разрешен, система продолжит проведение транзакции.
     * `{"code": 10}` — Неверный номер заказа (платеж отклоняется).
     * `{"code": 12}` — Неверная сумма заказа (платеж отклоняется).
     * `{"code": 13}` — Платеж не может быть принят (общая ошибка, отклоняется).
     * `{"code": 20}` — Платеж просрочен (отклоняется).

2. **`Pay`**
   * Выполняется сразу после успешного прохождения авторизации/списания.
   * **Назначение:** Фиксация оплаты на стороне вашей системы (изменить статус заказа на "Оплачен", выдать доступ и т.д.). Содержит `Token` карты для рекарринга, если был передан `AccountId`.
   * **Ожидаемый ответ:** `{"code": 0}`.

3. **`Fail`**
   * Выполняется в случае неуспешного прохождения платежа (недостаточно средств, ошибка 3DS, карта заблокирована).
   * **Назначение:** Анализ причин отказов, вывод уведомления клиенту для повторной попытки.
   * **Ожидаемый ответ:** `{"code": 0}`.

4. **`Confirm`**
   * Выполняется при подтверждении двухстадийной операции по API или в ЛК.

5. **`Refund`**
   * Выполняется при осуществлении возврата средств.

6. **`Recurrent` / `Cancel`**
   * Выполняются при создании, списании или отмене подписок.

## 7. Рекуррентные платежи (Подписки)

Позволяет автоматически списывать средства с привязанных карт по заданному графику. 

### Шаги интеграции
1. Совершить первый успешный платеж с передачей параметров `accountId` и получением согласия пользователя на автосписания.
2. Из ответа API или webhook `Pay` сохранить параметр `Token`.
3. Создать подписку по API через метод `/subscriptions/create`.

### Параметры создания подписки (`POST /subscriptions/create`)

| Параметр | Тип | Описание | Обязательность |
| :--- | :--- | :--- | :--- |
| `Token` | string | Токен привязанной карты | **Да** |
| `AccountId` | string | Идентификатор пользователя в вашей системе | **Да** |
| `Description` | string | Описание подписки (отображается в ЛК и квитанциях) | **Да** |
| `Amount` | number | Сумма регулярного списания | **Да** |
| `Currency` | string | Валюта подписки | **Да** |
| `Interval` | string | Интервал списаний: `Day`, `Week`, `Month` | **Да** |
| `Period` | int | Периодичность (например, `Interval: Month`, `Period: 1` — раз в месяц) | **Да** |
| `StartDate` | DateTime | Дата и время первого автосписания в формате UTC (в будущем) | **Да** |
| `RequireConfirmation` | boolean | Использовать двухстадийную схему при автосписаниях | **Да** |
| `MaxPeriods` | int | Максимальное число списаний, после которых подписка завершится | Нет |

### Методы управления подписками
* **Получение информации:** `POST /subscriptions/get` (передается `Id` подписки).
* **Поиск подписок пользователя:** `POST /subscriptions/find` (передается `accountId`).
* **Изменение подписки:** `POST /subscriptions/update` (позволяет менять сумму, дату следующего списания, периодичность).
* **Отмена подписки:** `POST /subscriptions/cancel` (останавливает дальнейшие списания).

## 8. Альтернативные методы оплаты

CloudPayments поддерживает современные платежные методы в один клик без ввода данных карты.

### 1. СБП (Система Быстрых Платежей)
Позволяет оплачивать через сканирование QR-кода или перенаправление в приложение банка.

* **Получение платежной ссылки:** `POST /payments/qr/sbp/link`
  * Запрос содержит `PublicId`, `Amount`, `Currency: "RUB"`, `Scheme: "charge"`.
  * Ответ содержит `QrUrl` — ссылка для перенаправления пользователя или генерации собственного QR-кода.
* **Получение QR-кода в виде картинки:** `POST /payments/qr/sbp/image`
  * Возвращает изображение QR-кода в формате PNG, закодированное в Base64 (`QrImage`).
* **Список банков-участников СБП:** `POST /sbp/v2/banks/info`
  * Возвращает список банков с их логотипами (`logo`) и схемами открытия приложений (`url`). Используется для вывода красивого списка банков на мобильных устройствах.

### 2. SberPay
* **Получение платежной ссылки:** `POST /payments/qr/sberpay/link`
  * Возвращает `QrUrl` вида `sberpay://invoicing/v2...` для перенаправления в приложение СберБанк Онлайн.
* **Получение QR-кода для десктопа:** `POST /payments/qr/sberpay/image`
  * Возвращает QR-код в формате SVG, закодированный в Base64 (`QrImage`).

### 3. T-Pay (Tinkoff Pay)
* Интегрируется аналогично СБП/SberPay через специализированные эндпоинты для генерации платежных ссылок и QR-кодов.

### 4. Долями (BNPL-сервис)
Позволяет клиенту разделить сумму покупки на 4 части с оплатой каждые 2 недели. ТСП получает всю сумму сразу на следующий рабочий день.
* **Создание сессии:** `POST` с параметром `AltPayType: "TcsBnplDolyame"`
* **Ответ шлюза:** Содержит `ExtensionData.Link` — адрес страницы оформления рассрочки на сайте "Долями".

## 9. Онлайн-касса и 54-ФЗ (CloudKassir)

Сервис **CloudKassir** позволяет фискализировать платежи и отправлять электронные чеки в налоговую (ФНС) через ОФД в полном соответствии с Федеральным законом №54-ФЗ.

### Формат передачи позиций в объекте чека `receipt`

При вызове виджета или API оплаты вы можете передать объект `receipt`, структура которого стандартизирована под требования ФФД (Формат фискальных документов):

```json
{
  "Items": [
    {
      "label": "Наименование товара или услуги", // Макс. 128 символов
      "price": 500.00, // Цена за единицу товара
      "quantity": 2.00, // Количество
      "amount": 1000.00, // Итоговая сумма по позиции
      "vat": 20, // Ставка НДС: 20, 10, 0 (НДС 0%), -1 (Без НДС), 110 (НДС 10/110), 120 (НДС 20/120)
      "method": 0, // Признак способа расчета (тег 1214): 0 - Предоплата 100%, 3 - Аванс, 4 - Полный расчет
      "object": 1, // Признак предмета расчета (тег 1212): 1 - Товар, 4 - Услуга, 10 - Платеж
      "measurementUnit": "шт" // Единица измерения
    }
  ],
  "taxationSystem": 0, // Система налогообложения (тег 1055): 0 - ОСН, 1 - УСН Доход, 2 - УСН Доход-Расход, 3 - ЕНВД, 4 - ЕСХН, 5 - ПСН
  "email": "customer@example.com", // Email покупателя для отправки чека
  "phone": "+71234567890", // Номер телефона для СМС со ссылкой на чек
  "amounts": {
    "electronic": 1000.00 // Сумма оплаты электронными деньгами (должна совпадать с общей суммой платежа)
  }
}
```

## 10. Справочники и коды ошибок

### Статусы транзакций
* **`AwaitingAuthentication`** — Ожидает подтверждения 3-D Secure со стороны держателя карты.
* **`Authorized`** — Двухстадийный платеж успешно авторизован и сумма заблокирована (ожидает подтверждения или отмены).
* **`Completed`** — Оплата успешно завершена, денежные средства списаны.
* **`Cancelled`** — Операция успешно отменена (актуально для Authorized платежей).
* **`Declined`** — Операция отклонена шлюзом или банком-эмитентом.

### Статусы подписок (рекуррент)
* **`Active`** — Подписка активна, списания проводятся по графику.
* **`PastDue`** — Просрочена (после 1 или 2 неудачных попыток списания шлюз будет повторять попытки).
* **`Cancelled`** — Отменена ТСП или пользователем.
* **`Rejected`** — Отклонена окончательно (после 3 неудачных попыток подряд списания прекращаются).
* **`Expired`** — Завершена по достижении максимального числа периодов (`MaxPeriods`).

### Группы ошибок эквайринга
Шлюз возвращает коды ошибок в случае неудачных платежей. Их важно правильно обрабатывать для рекуррентных списаний:

1. **Безусловно негативные** (Повторные списания строго запрещены):
   * `5005` (Do Not Honor — Отказ эмитента без указания причин).
   * `5057` (Transaction Not Permitted — Транзакция запрещена для этой карты).
   * `5001` (Refer To Card Issuer — Свяжитесь с банком).
   * `5054` (Expired Card — Карта просрочена).
   * `5034` (Suspected Fraud — Подозревается в мошенничестве).
2. **Условно/Временно негативные** (Можно попробовать списание позже или через 24 часа):
   * `5051` (Insufficient Funds — Недостаточно средств на карте).
   * `5096` (System Error — Временный сбой на стороне банка/платежной системы).
   * `5065` (Exceed Withdrawal Frequency — Превышен лимиты частоты использования карты).
