# Модуль базы знаний: обзор, позиционирование и материалы для лендинга / FAQ

## 1. Что это за модуль

Модуль базы знаний в DashAdmin - это внутренняя система инструкций, регламентов и операционных материалов для клуба.

Это не просто набор заметок. Модуль связывает между собой:

- разделы и подкатегории;
- статьи и инструкции;
- поиск по содержимому;
- права на просмотр и управление;
- шаблоны типовых статей;
- единый интерфейс для управляющих и сотрудников.

Идея модуля: знания клуба не должны жить в чатах, голосовых, Google Docs и памяти отдельных людей.

Не просто "где-то был текст про уборку кресла", а:

- статья лежит в нужном разделе;
- ее можно быстро найти по поиску;
- у нее есть понятный заголовок;
- ее можно открыть и прочитать с телефона;
- управляющий может обновить инструкцию прямо в системе;
- сотрудники видят актуальную версию в рабочем интерфейсе клуба.

## 2. Для кого этот модуль

Модуль рассчитан на клубы и команды, где важно быстро передавать операционные знания без хаоса.

Основные роли внутри модуля:

- владелец клуба;
- управляющий;
- администратор;
- сотрудник смены;
- техперсонал;
- новые сотрудники на онбординге.

Это особенно полезно там, где есть повторяемые процессы:

- открытие и закрытие смены;
- обслуживание и уборка;
- реакции на инциденты;
- работа с гостями;
- сетевые и технические регламенты;
- стандарты работы для разных залов и смен.

## 3. Какую проблему он решает

Без нормальной базы знаний знания в клубе обычно размазаны по разным каналам:

- старые сообщения в чатах;
- заметки в телефоне;
- файлы без структуры;
- устные объяснения от смены к смене;
- "спроси у того, кто давно работает".

Из-за этого появляются типовые проблемы:

- сотрудник не знает, где искать инструкцию;
- один и тот же вопрос задают снова и снова;
- стандарты обслуживания плавают между сменами;
- новые сотрудники дольше входят в работу;
- критичные действия при сбоях зависят от памяти конкретного человека;
- важные знания теряются при смене команды.

Модуль закрывает это за счет простой операционной модели:

- знания хранятся централизованно;
- статьи разложены по категориям и подкатегориям;
- поиск работает по заголовку и содержимому;
- сотрудники видят материалы в читаемом виде;
- управляющие могут быстро актуализировать инструкции;
- типовые статьи можно создавать из шаблонов.

## 4. Из чего состоит модуль

### 4.1. Основной экран базы знаний

Основной экран дает две базовые вещи:

- навигацию по структуре знаний;
- быстрый поиск по статьям и категориям.

На desktop это боковая структура разделов и область чтения.
На mobile это переработанный flow, где стартовый экран сразу показывает:

- заголовок раздела;
- поиск;
- список категорий и статей;
- быстрый вход в нужную инструкцию без скрытого пустого экрана.

Это важно для позиционирования: база знаний должна работать как рабочий инструмент, а не как архив документов.

### 4.2. Категории и подкатегории

База знаний строится вокруг дерева разделов.

Что уже есть:

- категории;
- вложенные подкатегории;
- порядок отображения;
- описание раздела;
- быстрый переход в нужный блок знаний;
- создание подкатегорий прямо из выбранного раздела.

Это позволяет строить структуру не по файлам, а по логике клуба:

- открытие смены;
- уборка;
- техника;
- сеть;
- касса;
- работа с конфликтами;
- экстренные ситуации.

### 4.3. Статьи

Статья - это основной контейнер знаний в модуле.

Что уже есть:

- отдельный заголовок;
- rich-text содержимое;
- дата обновления;
- автор в просмотре статьи;
- привязка к конкретной категории;
- чтение в отдельной области контента;
- редактирование из интерфейса без перехода во внешние сервисы.

Важный момент: статья живет внутри рабочей структуры клуба, а не как изолированная заметка.

### 4.4. Поиск

Поиск работает не только по названиям, но и по содержимому статей.

Это значит:

- можно искать по ключевому слову;
- можно находить статьи, даже если не помнишь раздел;
- категории тоже подстраиваются под результат поиска;
- дерево не превращается в шум, а показывает только релевантные разделы.

Для повседневной операционки это критично, потому что сотрудник чаще помнит проблему или слово из инструкции, чем точную папку.

### 4.5. Редактор знаний

В модуле есть встроенный rich-text редактор.

Что он уже поддерживает:

- заголовки;
- обычный текст;
- жирный, курсив, underline, strike;
- inline code и code block;
- цитаты;
- маркированные и нумерованные списки;
- ссылки;
- изображения;
- выравнивание;
- цвет текста;
- цвет выделения;
- горизонтальные разделители;
- undo / redo.

Также сейчас уже улучшен mobile editing UX:

- действия `Отмена / Сохранить` вынесены в sticky bottom;
- toolbar редактора сделан компактнее;
- на мобильных он прокручивается горизонтально;
- под сам текст остается больше полезного пространства.

### 4.6. Шаблоны статей

В системе уже есть шаблоны для типовых материалов.

Сейчас в коде зашиты примеры для:

- запуска игровых ПК;
- действий при сбое сети;
- стандарта обслуживания зоны.

Это важно для коммуникации, потому что база знаний здесь не только "хранит текст", но и помогает быстрее запускать типовые инструкции.

### 4.7. Права доступа

У модуля есть разделение на просмотр и управление.

Что видно по текущей реализации:

- сотрудники клуба могут читать KB;
- owner и управленческие роли могут создавать и редактировать структуру;
- в UI также учитывается право `manage_kb`.

Это значит, что база знаний работает как единый источник правил для всех, но редактируется ограниченным кругом людей.

### 4.8. Employee-side сценарий

У базы знаний есть отдельный employee-view.

Это значит:

- материалы доступны не только в админской части;
- сотрудник может открыть статью в рабочем интерфейсе;
- знания можно использовать прямо в смене, а не только при настройке системы.

Для лендинга это сильный пункт: база знаний не заперта на управляющем, она реально используется в операционке.

## 5. Ключевая бизнес-ценность

### 5.1. Для владельца и управляющего

- Знания клуба перестают зависеть от памяти отдельных людей.
- Проще держать единый стандарт между сменами и филиалами.
- Новые инструкции можно быстро доводить до команды.
- Меньше повторяющихся ошибок из-за "я не знал, как правильно".

### 5.2. Для операционной команды

- Не нужно искать инструкции в чатах и старых файлах.
- Проще ориентироваться в регламентах по разделам.
- Поиск экономит время в реальной смене.
- Доступ с телефона делает систему пригодной не только для офиса, но и для живой операционки.

### 5.3. Для онбординга и качества

- Новый сотрудник быстрее входит в процессы.
- Стандарты можно объяснять через готовые статьи, а не устно каждый раз.
- Снижается разброс в исполнении одних и тех же задач.
- Клуб получает централизованную базу регламентов, а не набор разрозненных текстов.

## 6. Что особенно сильное в этой реализации

Это то, что реально можно выносить в коммуникацию.

### 6.1. Это не файловое хранилище, а структурированная база знаний

Система строится вокруг дерева категорий и статей, а не вокруг папок с документами.

Это упрощает:

- навигацию;
- поиск;
- поддержку структуры;
- масштабирование по мере роста клуба.

### 6.2. База знаний встроена в рабочий контур клуба

Это не отдельный wiki-сервис, куда никто не заходит.

Материалы лежат внутри DashAdmin, рядом с остальными операционными модулями.
Это повышает шанс, что ими реально будут пользоваться.

### 6.3. Есть разделение между чтением и управлением

Это важно для реальной эксплуатации:

- сотрудники читают актуальные инструкции;
- редактирование не открыто всем подряд;
- управляющая команда поддерживает порядок и актуальность базы.

### 6.4. Mobile UX имеет практический смысл

База знаний читается и используется в телефоне:

- можно найти статью на смене;
- можно открыть инструкцию прямо у рабочего места;
- можно редактировать материалы без тяжелого desktop-only интерфейса.

### 6.5. Есть готовая база для стандартизации

Шаблоны и структура помогают клубу не просто хранить знания, а формализовать повторяемые процессы.

## 7. Основные сценарии использования

### Сценарий 1. Онбординг нового сотрудника

1. Управляющий создает базовую структуру разделов.
2. Внутрь добавляются ключевые статьи по смене, обслуживанию и технике.
3. Новый сотрудник получает доступ к базе знаний.
4. Вместо устных пересказов он читает актуальные материалы в системе.

### Сценарий 2. Работа в текущей смене

1. Сотрудник сталкивается с нестандартной ситуацией.
2. Через поиск находит нужную инструкцию.
3. Открывает статью на телефоне или в рабочем интерфейсе.
4. Действует по актуальному регламенту.

### Сценарий 3. Обновление стандартов

1. Управляющий замечает, что текущий процесс нужно поменять.
2. Открывает нужную статью.
3. Обновляет текст через встроенный редактор.
4. Команда начинает использовать новую версию без рассылок по чатам и новых файлов.

### Сценарий 4. Создание типовой инструкции

1. Выбирается категория.
2. Создается новая статья.
3. При необходимости используется шаблон.
4. Статья дорабатывается под процессы конкретного клуба.

### Сценарий 5. Масштабирование на сеть

1. Формируется единая структура знаний.
2. В базу заносятся стандарты по технике, обслуживанию и сменам.
3. Разные филиалы работают по одной логике.
4. Управление знаниями перестает зависеть от локальных привычек отдельных команд.

## 10. FAQ-заготовки

### Что умеет модуль базы знаний?

Модуль позволяет создавать структуру разделов и подкатегорий, вести статьи и инструкции, искать материалы по заголовкам и содержимому, использовать шаблоны и открывать знания внутри рабочего интерфейса клуба.

### Это просто раздел с текстовыми заметками?

Нет. Это структурированная база знаний с категориями, поиском, встроенным редактором и доступом для сотрудников. Она рассчитана на ежедневное использование в операционке, а не на хранение случайных заметок.

### Можно ли создавать вложенные разделы?

Да. База знаний поддерживает категории и подкатегории, что позволяет строить понятную структуру по процессам, ролям или темам.

### Можно ли быстро найти нужную инструкцию?

Да. Поиск работает по названиям и содержимому статей, поэтому можно искать не только по структуре, но и по ключевым словам.

### Кто может редактировать материалы?

По текущей логике модуль различает просмотр и управление. Сотрудники могут читать материалы, а owner и управленческие роли управляют структурой и содержимым. В интерфейсе также учитывается право `manage_kb`.

### Можно ли пользоваться базой знаний с телефона?

Да. Мобильная версия уже адаптирована под чтение, навигацию и редактирование, чтобы знания были доступны прямо в процессе работы.

### Есть ли шаблоны статей?

Да. В модуле уже есть шаблоны для типовых материалов, например для запуска игровых ПК, действий при сбое сети и стандартов обслуживания зоны.

### Подходит ли база знаний только для технических инструкций?

Нет. В ней можно хранить любые рабочие материалы клуба: регламенты смены, стандарты общения, правила уборки, инструкции по технике, сценарии реагирования на сбои и другие внутренние процессы.

### Можно ли использовать модуль для онбординга?

Да. Это один из самых сильных сценариев. База знаний помогает быстро вводить новых сотрудников в процессы через готовую структуру статей и регламентов.

### Зачем хранить знания внутри DashAdmin, если есть Google Docs или Notion?

Потому что знания оказываются прямо внутри рабочего контура клуба. Сотрудникам не нужно переключаться между внешними сервисами, а управляющей команде проще поддерживать единый порядок доступа и структуры.

### Можно ли открыть базу знаний сотрудникам, но ограничить редактирование?

Да. Это как раз нормальная модель использования: команда читает статьи, а редактирование остается у ответственных ролей.
